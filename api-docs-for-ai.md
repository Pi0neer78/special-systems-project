# API СпецСистемы — Документация для разработки Android-приложения

## Назначение приложения

Android-приложение для сотрудников компании СпецСистемы. Сотрудники обрабатывают заявки от клиентов: просматривают список, берут в работу, назначают ответственных, фиксируют результат, меняют статус.

---

## Общие сведения

**Base URL:** `https://functions.poehali.dev/4866cc97-c798-42d4-a280-d35071d704a8`

**Формат данных:** JSON  
**Content-Type для запросов с телом:** `application/json`  
**Кодировка:** UTF-8

**Авторизация:** все запросы (кроме `staff-login`) требуют заголовок:
```
X-Admin-Token: <token>
```

Токен получается при входе (`staff-login`). Действует ~2 часа. Хранить в `SharedPreferences`. При каждом старте приложения проверять через `staff-verify` — если ответ 401, перенаправить на экран входа.

## Справочники (константы)

### Приоритеты

| Значение API | Отображение в UI |
|---|---|
| `low` | Низкий |
| `medium` | Средний |
| `high` | Высокий |
| `urgent` | Срочно |

### Статусы

| Значение API | Отображение в UI |
|---|---|
| `new` | Новая |
| `in_progress` | В процессе |
| `resolved` | Решена |
| `cancelled` | Отменена |

### Типы проблем

```
"Вопрос по 1С"
"Проблема с доступом"
"Нужно обновление"
"Ошибка при работе"
"Нужна доработка"
"Нужна консультация"
"Проблемы с оборудованием"
"Прочее"
```

## Модель данных заявки (Ticket)

```
id                : Int       — уникальный ID
client_id         : Int       — ID клиента
client_name       : String    — название организации клиента
submitted_at      : String    — дата подачи, ISO 8601 ("2026-06-20T09:15:00+03:00")
priority          : String    — low | medium | high | urgent
problem_type      : String    — тип проблемы (строка из справочника)
description       : String    — описание проблемы
deadline          : String?   — срок решения, ISO 8601, может быть null
extra_info        : String?   — дополнительная информация, может быть null
result            : String?   — результат работы (заполняет сотрудник), может быть null
status            : String    — new | in_progress | resolved | cancelled
resolved_at       : String?   — дата решения ISO 8601, заполняется автоматически, может быть null
status_changed_at : String    — дата последнего изменения статуса, ISO 8601
assignee_id       : Int?      — ID ответственного сотрудника, может быть null
assignee_name     : String?   — полное имя ответственного, может быть null
assignee_login    : String?   — логин ответственного, может быть null
```

**Просроченная заявка** — условие: `resolved_at == null` И текущая дата/время > `deadline`. Выделять красным цветом в списке и на карточке.

## Методы API

---

### 1. Вход сотрудника

**POST** `?resource=staff-login`

Авторизация по логину и паролю. Токен сохранить в SharedPreferences.

**Заголовки:**
```
Content-Type: application/json
```

**Тело запроса:**
```json
{
  "login": "ivanov",
  "password": "mypassword"
}
```

**Успешный ответ — 200:**
```json
{
  "token": "a3f9c2e1d4b7083c9e2f1a6d5b8c0e3f7a2d9b4c1e8f0a3d6b9c2e5f8a1d4b7",
  "user_id": 5,
  "login": "ivanov",
  "full_name": "Иванов Иван Иванович",
  "role": "user"
}
```

> `role` = `"user"` (обычный сотрудник) или `"admin"` (администратор). Для Android-приложения роль влиять на UI не должна — оба типа могут работать с заявками одинаково.

**Ошибка — 400:**
```json
{ "error": "Укажите логин и пароль" }
```

**Ошибка — 401:**
```json
{ "error": "Неверный логин или пароль" }
```

---

### 2. Проверка токена при старте приложения

**GET** `?resource=staff-verify`

Вызывать при каждом запуске приложения перед показом главного экрана.

**Заголовки:**
```
X-Admin-Token: <token>
```

**Успешный ответ — 200:**
```json
{
  "ok": true,
  "user_id": 5,
  "login": "ivanov",
  "full_name": "Иванов Иван Иванович",
  "role": "user"
}
```

**Ошибка — 401:**
```json
{ "ok": false, "error": "Токен недействителен" }
```

> Если 401 — удалить токен из SharedPreferences и открыть экран входа.

### 6. Редактирование заявки

**PATCH** `?resource=tickets&id=42`

Все поля тела опциональны — передавать только то, что нужно изменить.

**Заголовки:**
```
X-Admin-Token: <token>
Content-Type: application/json
```

**Поля тела запроса:**

| Поле | Тип | Допустимые значения | Описание |
|---|---|---|---|
| `status` | string | `new`, `in_progress`, `resolved`, `cancelled` | Статус заявки |
| `assignee_id` | number \| null | ID из справочника users или `null` | Ответственный сотрудник |
| `result` | string \| null | Любой текст или `null` | Результат работы |
| `priority` | string | `low`, `medium`, `high`, `urgent` | Приоритет |
| `problem_type` | string | Строка из справочника | Тип проблемы |
| `deadline` | string \| null | ISO 8601 или `null` | Срок решения |
| `extra_info` | string \| null | Любой текст или `null` | Доп. информация |

> При `status: "resolved"` сервер автоматически устанавливает `resolved_at = текущее время`.  
> При смене статуса с `resolved` на любой другой — `resolved_at` сбрасывается в `null`.

**Пример 1 — взять заявку в работу и назначить себя ответственным (user_id = 5):**

Запрос:
```json
{
  "status": "in_progress",
  "assignee_id": 5
}
```

Ответ (200):
```json
{
  "id": 42,
  "status": "in_progress",
  "assignee_id": 5,
  "assignee_name": "Иванов Иван Иванович",
  "assignee_login": "ivanov",
  "status_changed_at": "2026-06-20T10:00:00+03:00",
  "resolved_at": null,
  ...
}
```

**Пример 2 — закрыть заявку с результатом:**

Запрос:
```json
{
  "status": "resolved",
  "result": "Переустановили платформу 1С версии 8.3.24. Клиент подтвердил работоспособность."
}
```

Ответ (200):
```json
{
  "id": 42,
  "status": "resolved",
  "result": "Переустановили платформу 1С версии 8.3.24. Клиент подтвердил работоспособность.",
  "resolved_at": "2026-06-20T17:30:00+03:00",
  "status_changed_at": "2026-06-20T17:30:00+03:00",
  ...
}
```

**Пример 3 — отменить заявку:**

Запрос:
```json
{
  "status": "cancelled"
}
```

**Пример 4 — изменить приоритет и срок:**

Запрос:
```json
{
  "priority": "urgent",
  "deadline": "2026-06-21T12:00:00+03:00"
}
```

**Пример 5 — снять ответственного:**

Запрос:
```json
{
  "assignee_id": null
}
```

**Ошибки:**
```json
{ "error": "Заявка не найдена" }   // 404
{ "error": "Нечего обновлять" }    // 400 — пустое тело {}
{ "error": "Не авторизован" }      // 401 — неверный токен
{ "error": "Недостаточно прав" }   // 403 — клиент не может редактировать
```

## Обработка ошибок

| HTTP статус | Значение | Действие |
|---|---|---|
| `200` | Успех | Обработать данные |
| `201` | Создано | Обработать данные |
| `400` | Ошибка запроса | Показать `error` пользователю |
| `401` | Токен недействителен | Удалить токен, открыть экран входа |
| `403` | Нет прав | Показать сообщение об ошибке |
| `404` | Не найдено | Показать сообщение об ошибке |
| `405` | Неверный метод/ресурс | Ошибка разработки |

Все ошибки возвращают JSON: `{ "error": "текст ошибки" }`

## Рекомендуемая архитектура приложения

### Экраны

1. **Экран входа** — поля логин/пароль, кнопка «Войти»
2. **Список заявок** — с фильтрами по статусу, клиенту, ответственному; отметка просроченных
3. **Карточка заявки** — все поля, кнопки быстрых действий
4. **Редактирование заявки** — форма со всеми полями

### Сценарий запуска

```
Запуск приложения
  → Есть токен в SharedPreferences?
      НЕТ  → Экран входа
               → POST staff-login
               → Сохранить token, user_id, full_name в SharedPreferences
               → Загрузить справочники (GET ticket-meta)
               → Перейти на список заявок
      ДА   → GET staff-verify
               → 200 ok:true  → Загрузить справочники → Список заявок
               → 401          → Удалить токен → Экран входа
```

### Сценарий работы с заявкой

```
Список заявок (GET tickets)
  → Нажать на заявку
  → Карточка заявки (GET tickets&id=N)
  → Нажать "Редактировать"
  → Форма редактирования
  → Сохранить (PATCH tickets&id=N)
  → Обновить карточку / вернуться в список
```

### Быстрые действия из списка (свайп или кнопки)

- «Взять в работу» → `PATCH { "status": "in_progress", "assignee_id": <свой user_id> }`
- «Решена» → `PATCH { "status": "resolved" }`
- «Отменить» → `PATCH { "status": "cancelled" }`
